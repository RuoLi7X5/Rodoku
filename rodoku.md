# Rodoku（逻辑型数独 AI）开发需求与路径方案报告

> 本报告依据 `AItaolun.md` 中用户与 AI 的三轮对话整理，目标是把“只设定底层规则，让系统自我进化”的设想，落到**可实现、可迭代、可验证**的工程路径上，并显式规避“捷径学习/暴力猜测/回溯式秒解”的偏离。

---

## 1. 背景与核心诉求（从对话抽取）

- **项目愿景**：打造一套数独求解系统（Rodoku），像 AlphaGo Zero 一样在只给定规则的情况下，通过大量重复解题/探索路径自我成长与迭代。
- **关键差异点**：用户不希望它变成“回溯/穷举/直接填正确答案”的求解器，而希望它像人类高手一样：
  - 依赖**提示数**与**候选数（pencil marks）**之间的关系；
  - 通过**逻辑排除**逐步收敛；
  - 最终在“证据充分”时**确认填数**。
- **对抗目标（明确要求禁止）**：  
  - 仅为得分而“猜测填数”或隐式学习捷径（Shortcut Learning）；
  - 用传统回溯/DLX 作为主体解题路径（即便很快、很准，但与目标不符）。
- **硬件现实**：本地仅办公本（i5-13500H 等 CPU），足够开发与验证，但不适合大规模训练；训练倾向国内云算力，预算期望 **≤ 2 元/小时**（按平台实际为准）。

---

## 2. 目标定义（必须说清楚“要什么/不要什么”）

### 2.1 产品级目标（What）

- **逻辑型求解**：输出不仅是答案，还应包含**逐步推理轨迹**（至少是“候选数被排除/某格被确定”的序列）。
- **可训练可进化**：能在题库上持续训练，整体表现（成功率、步数、违规率等）可度量并提升。
- **可视化可审计**：支持回放“它为什么排除/为什么确认”的过程，便于人类验证其“非偷懒”。

### 2.2 研究/工程边界（Non-goals，先不做）

- **不以“秒解”为主 KPI**：速度不是第一优先级，正确且可解释的逻辑链路更重要。
- **不追求一次覆盖全部高级技巧**：先从基础可学的排除模式与约束传播起步，逐步扩展到更复杂技巧。
- **不先上 MCTS**：对话建议先把“严苛环境规则”写好，MCTS 属于进阶增强项。

## 3. 需求清单（开发需求整理）

### 3.1 功能需求（Functional）

- **题目导入**
  - 只聚焦 **9×9 标准数独（9宫）**。
  - 支持从题库加载文本格式数独：空格用 `0` 或 `.`；题与题之间用分隔（如空行/分隔符）隔开。
  - 可按难度/空格数做课程学习（curriculum）分桶。
- **规则引擎（裁判）**
  - 检查行/列/宫冲突。
  - 维护候选数集合，并在填入后进行约束传播更新。
  - 能判断：动作是否合法、是否导致矛盾、是否完成。
- **状态表示（Observation）**
  - 必须包含候选信息：典型形态为 **9x9x9 布尔/0-1 张量**（每格对 1..9 的可行性）。
  - 可选附加通道：已填数字 one-hot、给定提示（givens）掩码、矛盾标记等。
- **动作空间（Action）**（强制逻辑流）
  - 动作 A：**排除** `Eliminate(x, y, d)`：声称该格不可能是 d。
  - 动作 B：**确认/填入** `Commit(x, y, d)`：确认该格为 d。
  - 必须配置**强约束**：所有 Eliminate/Commit 都必须满足“可被验证的逻辑依据”（见 4.3），以彻底封死猜测。
- **奖励机制（Reward）**（指挥棒）
  - 排除正确的候选：小正奖（鼓励推进）。
  - 合法且有依据地确认填数：较大正奖（鼓励收敛）。
  - 排除掉真实解/造成矛盾：大惩罚并终局（强烈禁止）。
  - 未满足证据条件的 Commit（等价于赌博猜测）：显著惩罚（封死捷径）。
  - 重复/无效动作：惩罚（避免循环与刷分）。
- **训练与评估**
  - 训练循环可在云端运行；本地可跑小规模验证。
  - 输出指标与日志：成功率、平均步数、违规率、平均每局 reward、候选减少速度等。
- **可视化**
  - 展示候选数消失过程（颜色/标记）。
  - 支持回放每一步动作与即时“依据/证明片段”（例如“假设 r3c4=7 则无解，因此排除 7”；或“r3c4 其他值均无解，因此确定=7”）。

### 3.2 非功能需求（Non-functional）

- **可复现**：固定随机种子、版本锁定、记录训练配置。
- **可解释/可审计**：每个 Eliminate/Commit 必须输出“依据（proof/rationale）”，且能被裁判验证为真；不要求自带“技巧标签”。
- **可扩展**：允许使用任意正确逻辑（不限技巧上限）。技巧“名字/标签”只能由人类事后映射，不作为智能体必须输出的内容。
- **鲁棒性**：题库不干净（多解/无解/格式异常）时要能报错或跳过。

## 4. 关键设计：如何“封死偷懒路径”

### 4.1 为什么必须把“候选数”作为状态

若仅输入 9x9 数字盘面，Agent 可能学到“直觉猜答案”并获得回报。  
将状态升级为 **9x9x9 候选张量**，让“排除过程”成为可学习对象，且便于审计其行为。

### 4.2 动作必须包含“排除”，而非只允许“填数”

只允许填数会天然诱导赌博式搜索。  
加入 Eliminate 并给予奖励，才能把学习目标从“终局正确”转移到“中间推理正确”。

### 4.3 Commit 的“证据门槛”（硬规则，而不是软奖励）

你明确要求：**不限制逻辑上限，但必须正向推理、禁止代入试数/回溯式试填**。因此这里的“证据门槛”要同时满足两点：

- **动作必须逻辑必然**（不是“填了再看会不会错”）
- **解释必须是“候选关系推导”**（不要求技巧标签，但要说清楚依据）

为做到“零猜测 + 可审计”，推荐采用 **“生成-验证分离”** 的硬约束：

- **生成（Agent 负责）**：提出一个动作（Eliminate/Commit）并给出“依据（rationale）”，依据形式是可机器校验的“候选关系推导记录”（不是技巧名）。
- **验证（Oracle 负责）**：用传统求解器/约束求解对动作做“逻辑必然性校验”，只作为裁判，不作为策略本身。

#### 4.3.1 动作正确性的严格定义（与你描述的正向推理一致）

- **Eliminate 合法**：该候选在当前盘面约束下**不可能成立**（逻辑上必然为假）。  
  直观等价于：无论后续怎么推，最终解里都不可能出现 `r,c=d`。
- **Commit 合法**：该格该值在当前盘面约束下是**必然值**（forced），即你描述的：
  - “这个格只剩这一个候选（单元格唯一）”，或
  - “这个数字在该行/列/宫里只可能落在这一个格（位置唯一）”，或
  - “由候选之间的矛盾关系推出其它候选全部被排除，因此该值必然成立”

关键点：**Commit 的定义不是“填进去试试再看冲突/不冲突”**，而是“在填之前就已被逻辑推出为必然”。

#### 4.3.2 Oracle 校验（防作弊，但不替代解释）

Oracle 的作用是“保证动作确实必然正确”，从机制上杜绝：
- 通过运气/试数得到正确答案
- 通过隐式搜索得到结论但解释胡编

校验方式（实现可选其一）：
- **求解器蕴含校验**：检查动作是否被当前约束系统逻辑蕴含（例如候选不可行/该值必然）
- **反证校验（内部可用）**：例如验证“若 `r,c=d` 则无解”，用于确认 Eliminate 的正确性  

> 重要：你强调“不希望代入试数”。因此在**对外解释**里，不强制要求暴露“假设→矛盾”的反证形式；解释更建议输出为“候选关系推导链”（例如类似你举的‘行内两格仅{1,3}’导致其它格删{1,3}）。Oracle 只做后台裁判。

## 5. 环境（RodokuEnv）规格草案（工程可落地）

### 5.1 状态（Observation）

最低配置（推荐）：
- `candidates[9][9][9]`：布尔/0-1，表示该格该数是否仍可能。
- `given[9][9]`：是否为题目给定数（不可改）。
- `value[9][9]`：当前填入值（0 表示未填）。

可选增强：
- `invalid_mask`：标记当前是否存在冲突（用于调试，训练时一般应避免出现）。
- `step_budget`：剩余步数/回合步数（防止拖延）。

### 5.2 动作（Action）

离散化建议（便于 DQN/PPO 等）：
- 编码为整数：`a = ((type * 81 + (x*9+y)) * 9 + d)`  
  - `type ∈ {0=Eliminate, 1=Commit}`
  - `x,y ∈ [0..8]`
  - `d ∈ [0..8]` 对应数字 1..9

同时提供可读结构体形式，便于日志与可视化。

### 5.3 转移与合法性检查（Step）

每一步需执行：
- **动作解码**
- **动作合法性**
  - 给定格不可操作
  - Eliminate：候选存在才有效；若已排除则判为重复/无效
  - Eliminate/Commit：必须通过 **Oracle 的“逻辑必然性校验”**（见 4.3）；否则判为“无依据动作”并惩罚/终局（视策略）
- **一致性维护**
  - Eliminate 后更新该格候选
  - Commit 后写入值，并触发约束传播（同屋候选剔除等）
- **终局判断**
  - 产生矛盾/排除真实解：失败终局
  - 全部填满且合法：成功终局
  - **无进度超时**：在设定窗口内没有任何“有效排除/有效填数”（例如候选总数不下降、已填格不增加），判失败（你要求：卡住不许猜测）
  - 超步数：失败或截断（按策略）

### 5.4 奖励函数（Reward）建议初值

（具体数值需通过实验调参，但方向必须明确）

- `+0.1`：成功排除一个“确实不可能”的候选（且非重复）
- `+10`：在证据门槛满足时 Commit 正确
- `-10`：排除掉真实解 / 引入矛盾（终局）
- `-5`：未通过 oracle-proof 的 Eliminate/Commit（无依据）
- `-1`：重复/无效动作（对刷与死循环）
- `+100`：成功完成全盘（终局奖励，可选）

> 说明：为避免稀疏奖励，排除与正确 Commit 的中间奖励是必须的；但更关键的是 4.3 的硬门槛。

## 6. 训练策略（从零可跑通的路线）

### 6.1 课程学习（Curriculum Learning）

分阶段逐步加难：
- 先用“空格极少（如 1~5 个空）”的题，让系统学会基本约束与唯一性门槛。
- 再逐步提升空格数/难度，观察成功率曲线是否能跟上。

### 6.2 算法选择建议（对话一致）

优先级建议：
- **先实现环境与规则**（最关键壁垒）
- 再接入 RL：
  - 入门：DQN（离散动作，工程简单）
  - 稳定：PPO（策略梯度，普遍更稳）

> 不建议一开始就上 MCTS；可在后续作为“规划增强层”，但仍需遵循证据门槛。

### 6.3 “标准答案/裁判”角色

需要一个传统求解器（回溯/DLX/SAT 任一）作为“裁判/oracle”：
- **数据筛选**：判断题目是否有解、是否唯一解（你已确认：题库保证唯一解，这对“必然填数/必然删候选”非常关键）。
- **动作验证**：
  - Eliminate：验证假设该值是否导致无解。
  - Commit：验证该值是否为逻辑必然（其否定无解）。

注意：它只能当“裁判/验证器/数据生成器”，不能当 Agent 的主策略，否则违背项目目标。

## 7. 评估体系（必须量化“有没有在讲逻辑”）

建议核心指标：
- **成功率（Solve Rate）**：在给定步数上限内完成比例
- **违规率（Gambling Rate）**：未满足门槛却尝试 Commit 的比例（应趋近 0）
- **无依据动作率（No-proof Rate）**：未通过 oracle-proof 的动作比例（应趋近 0）
- **无效动作率**：重复/对已排除候选再排除等
- **平均步数/每格操作数**：效率度量（但非首要 KPI）
- **候选熵下降曲线**：候选分布收敛速度（体现“排除推进”）
- **证据强度与类型（可选）**：不要求输出技巧标签，但可统计“证明方式/冲突规模”等，以衡量推理质量

建议可视化：
- 每局回放：动作序列 + 候选变化 + 触发证据
- 训练曲线：reward、成功率、违规率随迭代变化

## 8. 工程架构建议（本地开发 + 云端训练）

### 8.1 代码分层（建议）

- `core/`：数独表示、候选维护、约束传播、技巧/证据检测
- `env/`：RodokuEnv（step/reset/reward/termination）
- `agents/`：DQN/PPO 策略与训练脚本
- `data/`：题库加载、难度分桶、验证（唯一解/无解筛除）

### 8.2 运行形态

- 本地（CPU）：
  - 开发与单元测试
  - 环境正确性验证（随机动作/脚本动作）
  - 小规模训练 smoke test（确认能跑）
- 云端（GPU）：
  - 批量训练与参数搜索
  - 定期导出模型与日志

- 题库格式统一、加载器完成
- 传统求解器可验证“有解/唯一解”

### 里程碑 M1：候选系统与约束传播（1~2 周）
- 候选 9x9x9 初始化/更新正确

- 单元测试覆盖（至少：Naked Single/Hidden Single 的检测）

### 里程碑 M2：RodokuEnv（强约束版）完成（1~2 周）
- Eliminate/Commit 动作完整

### 里程碑 M3：RL 接入并跑通（1~2 周）
- DQN 或 PPO 任一算法跑通

### 里程碑 M4：可视化与“解释性报告”（持续迭代）
- 回放界面/导出报告（每一步证据）
- 扩展更多证据门槛（Pairs/Locked Candidates…）
- 评估更难题集的泛化能力

## 10. 风险清单与对策（避免走偏/卡死）

- **稀疏奖励导致学不会**：用中间奖励 + 课程学习；最关键是动作门槛与可学习的排除空间。
- **奖励被刷（刷排除/重复动作）**：对重复/无效动作惩罚；限制每格/每候选的操作次数；设置步数上限。
- **题库存在多解/无解**：训练前用裁判筛掉或标注；否则“真实解”判定会混乱。
- **“必然值”过少导致推进慢**：优先用唯一解题；引入更强的候选推导（但仍以 oracle-proof 验证）；并用“无进度超时”作为失败信号，让策略学习避免停滞。
- **动作空间过大导致训练困难**：可加 action mask（仅允许当前合法动作被采样），显著提升学习效率与稳定性。

## 11. 已确认需求（你已明确回复的结论）

1. **范围**：只做 **9×9（9宫）** 标准数独。
2. **题库**：文本格式；空格用 `0` 或 `.`；题与题之间有分隔。
3. **解释性输出**：必须输出“排除/填数的依据与过程”，不要求也不假设模型自带技巧标签；技巧名字只有在人类提供映射后才可能出现。
4. **允许逻辑**：不设技巧上限；只要符合数独底层规则、推理正确、有足够理由即可。可与秩理论接轨；现有项目秩查询限制在 R<3，但 Rodoku 训练不受此限制。
5. **卡住策略**：不允许猜测；设定“无进度超时”，超时判失败。
6. **训练平台**：倾向国内、连接稳定、算力强；预算期望 **≤ 2 元/小时**。
7. **集成形态**：与当前项目合并，新增前端页面，两个页面互不影响但可互相跳转。

## 12. 与现有 `Rank-Sudoku` 前端项目的合并方案（落地建议）

你当前仓库是 Vite + React 单页应用（无路由）。为实现“两页互不影响但可跳转”，建议引入路由并拆分为两套页面组件：

- **页面 A（现有）**：`/rank`（或 `/`）继续承载“秩理论研究”现有功能，不改其交互与状态结构。
- **页面 B（新增）**：`/rodoku` 承载 Rodoku 的“训练/推理回放/解释性输出”展示。

路由建议：
- 若未来部署在静态站点且不方便做服务端重写，优先用 `HashRouter`（避免刷新 404）。

## 14. 仍需你确认的 1 个关键边界（避免“正向推理”理解不一致）

你举的例子是“数对推出删候选”，属于典型正向推理。  
但还有一类推理在数独界常见：**forcing/反证链**（假设某候选为真，经约束传播推出矛盾，从而删该候选）。它不是回溯解题，但形式上包含“假设”。

### 14.1 已确认结论：允许 forcing/反证链，但必须“短且可审计”

你已确认：**允许** 这类“假设→矛盾→排除”的反证链，但要求解释尽量简洁；同时你指出**秩理论的删数逻辑本身已涵盖链的传递形式**，因此后续可优先用“秩结构/链传递”的语言来表达依据。

为避免 forcing 退化为搜索（变相试数），建议对 forcing 施加硬上限（可配置）：
- **链深度上限**：例如 ≤ 3~5 次传播层级（按你们秩逻辑的链传递粒度定义）
- **传播步数/推导条数上限**：例如 ≤ 50 条确定性推导
- **单步用时上限**：例如 ≤ 50~200ms（本地/云端可分别配置）
- **禁止分支树扩展**：forcing 只允许“单一假设线”推进到矛盾，不允许多分支枚举

### 14.2 forcing 的“简洁解释格式”（建议）

对外输出时不需要展示冗长传播细节，采用最短摘要即可（并提供可展开的详细日志给调试）：
- **目标**：删 `rXcY<>d`
- **依据（摘要）**：`假设 rXcY=d →（秩链传递/确定性传播）→ 矛盾（冲突位置/冲突数字）→ 因此排除 d`
- **可选附加**：引用对应的“秩结构 ID/Truth-Link 片段”作为人类可核对的证据锚点（不需要技巧名）
